# 缓存使用分析与优化建议

## 一、P4 缓存架构配置

当前项目配置：

| 参数 | 值 | 配置宏 |
|------|-----|--------|
| L1 I-Cache | 16KB × 2 (每核独立) | 4-way, line=64B |
| L1 D-Cache | 64KB × 2 (每核独立) | 2-way, line=64B |
| L2 Cache | 256KB (双核共享) | 8-way, line=128B |
| PSRAM | 16MB | XIP 启用 (代码可在此执行) |
| SPIRAM malloc 阈值 | 16KB | >16KB 分配至 PSRAM |
| 内部 SRAM 保留 | 32KB | 预留给系统 |

```
┌─────────────────────────────────────────────────────┐
│                      PSRAM 16MB                     │
│  (相机帧/预览缓冲/游戏缓冲/模型/边缘图 —— 全部在此)   │
└────────────────────┬────────────────────────────────┘
                     │ ~50 周期延迟
                     ▼
┌─────────────────────────────────────────────────────┐
│                 L2 Cache 256KB (共享)                │
│                    8-way, 128B line                  │
└──────┬──────────────────────────────┬───────────────┘
       │                              │
       ▼                              ▼
┌──────────────┐              ┌──────────────┐
│ Core 0       │              │ Core 1       │
│ L1 I 16KB    │              │ L1 I 16KB    │
│ L1 D 64KB    │              │ L1 D 64KB    │
└──────────────┘              └──────────────┘
```

**关键特性**：
- P4 **没有硬件 Cache 一致性协议**，两核写同一 cache line 需软件 spinlock 保护
- L1 为 write-back 模式，L2 为两核共享的统一层
- DMA (相机/显示) 直接读写 PSRAM，绕过 CPU Cache
- 内部 SRAM 的访问是否经过 L1 Cache 可配置（当前 `CONFIG_SOC_CACHE_INTERNAL_MEM_VIA_L1CACHE=y`）

## 二、各组件缓存使用分析

### 2.1 Camera Task (Core 0, prio 5)

```
V4L2 mmap 帧缓冲: 2 × 800×640×2 = 2MB (PSRAM, DMA 写入)
```

| 属性 | 值 |
|------|-----|
| **CPU 访问量** | **极低** — 只传递指针，不读像素 |
| **L1 D 占用** | 几乎为零 |
| **L2 占用** | 几乎为零 |
| **DMA 一致性问题** | esp_video 框架已处理（V4L2 驱动负责 cache flush） |

### 2.2 Display / LVGL Task (Core 0, prio 4)

```
预览双缓冲:    2 × 640×512×2  = 1.25MB (PSRAM)
游戏双缓冲:    2 × 640×640×2  = 1.56MB (PSRAM)
LVGL 内部缓冲: ~150KB (PSRAM)
```

| 属性 | 值 |
|------|-----|
| **访问模式** | 混合读写 — LVGL 布局计算 + 渲染 + 缓冲交换 |
| **L1 D 占用** | ~30-50KB (当前工作区：字型、样式、DMA 描述符) |
| **L2 竞争** | **中等** — 与 Core 1 共享 L2，LVGL 的纹理/字体缓存会占用 L2 |
| **DMA** | MIPI-DSI 从 PSRAM 读帧缓冲送显，不经 CPU Cache |

**每次 LVGL 刷新周期（30fps，33ms）**：
1. `lv_timer_handler()` 遍历脏区域
2. 对每个脏对象执行 draw task（填充/文字/图像）
3. Draw task 将像素写回帧缓冲
4. 显示控制器 DMA 读取帧缓冲

### 2.3 IMU Task (Core 0, prio 4)

```
卡尔曼滤波:  ~200 bytes 状态 (6 float × 3 轴 + 协方差矩阵)
队列元素:    12 bytes (imu_attitude_t)
```

| 属性 | 值 |
|------|-----|
| **访问模式** | I2C 寄存器读取 (~μs) + 浮点矩阵运算 (~10 条指令) |
| **L1 D 占用** | **几乎为零** — 全部数据 < 1KB，完全在 L1 内 |
| **L2 占用** | 无需 L2 |
| **缓存效率** | **100%** — 所有数据在 L1 命中 |

IMU Task 是本系统缓存效率最高的组件。

### 2.4 Main Control Task (Core 1, prio 2)

这是最复杂、缓存压力最大的任务，包含三个主要计算路径：

#### 2.4.1 游戏渲染 (GAME_FPS=30, 持续运行)

```
单帧写入量:   640×640×2 = 819KB (PSRAM, 游戏后端缓冲)
单帧读取量:   瓦片精灵 (~512KB) + 物体精灵 (~128KB) + 弹珠精灵 (~4KB)
```

每帧渲染流程：
1. **地板填充** — 对 409,600 像素逐个计算 hash 噪声 → 顺序写 819KB
2. **瓦片 blit** — 最多 1600 个 16×16 精灵 → 随机读 + 顺序写
3. **物体 blit** — 32×32 精灵边缘混合 → 随机读 + 读改写
4. **弹珠绘制** — 3D 球体每像素法线旋转 → 计算密集

| 属性 | 值 |
|------|-----|
| **L1 D 占用** | 瓦片/物体精灵的当前 blit 行 + 帧缓冲的 ~128B cache line |
| **L2 竞争** | **高** — 819KB 帧缓冲远大于 L2(256KB)，流式写频繁驱逐旧 line |
| **L2 命中率** | **低** — 精灵数据 > L2，帧缓冲流式写是 cache 的反模式 |

**核心问题**：640×640 帧缓冲 = 819KB。每个像素写入都分配一个新的 L2 cache line。写完一整帧后，前面的 cache line 全被驱逐。这是经典的 **streaming store** 模式，cache 不仅无益，反而有写分配（write-allocate）开销。

#### 2.4.2 边缘检测 Canny (按需触发)

```
输入:         400×320 RGB565 = 256KB (PSRAM, 降采样帧)
中间缓冲:     400×320 × 3 = 384KB (PSRAM, 灰度/Gx/Gy/方向)
输出:         400×320 = 128KB (PSRAM, 二值边缘图)
总处理:       5 遍扫描 × 128K 像素 = 640K 像素操作
```

| 属性 | 值 |
|------|-----|
| **访问模式** | 5 次顺序扫描（高斯模糊 → Sobel → NMS → 双阈值 → 滞后） |
| **L1 D 占用** | 当前扫描行的 3×3 窗口（~几十字节） |
| **L2 命中率** | **中等** — 顺序扫描具有空间局部性，但每遍数据量(128-256KB)仍接近 L2 容量 |
| **跨核影响** | 边检在 Core 1 执行，与 Core 0 的 LVGL 共享 L2 |

#### 2.4.3 目标检测 YOLO (按需触发，最重)

```
模型大小:     YOLO11N_320_S8_V1 ~2.5MB (PSRAM)
输入帧:      800×640 → 320×320 RGB888 ~300KB
预处理缓冲:  ~600KB (RGB565↔RGB888 转换)
中间张量:    多个 80×80/40×40/20×20 特征图 (PSRAM)
```

| 属性 | 值 |
|------|-----|
| **L1 D 占用** | 当前卷积层的输入 patch + 权重行（< 64KB） |
| **L2 命中率** | **极低** — 模型权重 2.5MB >> L2 256KB，每层卷积遍历所有权重 |
| **Cache 抖动** | **严重** — YOLO 运行期间持续驱逐 L2 中 Core 0 的数据 |

YOLO 推理是系统中最严重的 cache 压力源。但它是按需触发（用户点 Detect / 游戏开始），不是持续负载。

### 2.5 Physics Task (Core 1, prio 3)

```
弹珠状态:        ~100 bytes (marble_state_t + 中间变量)
24 圆周采样点:    ~200 bytes (预计算 cos/sin 表)
赛道碰撞查找:     ~100KB 赛道栅格 (PSRAM, 随机访问)
```

| 属性 | 值 |
|------|-----|
| **L1 D 占用** | 弹珠状态 + 采样表 → 完全在 L1 内 (< 1KB) |
| **L2 占用** | 赛道栅格的随机访问 → 24 个 cache line / tick (100Hz = 2400 lines/s) |
| **赛道查找模式** | 24 点圆周采样 + 中心点 → 空间局部性好（点聚集在弹珠周围） |

物理任务缓存效率高，因为：
- 弹珠状态、速度、法线全部在寄存器/L1
- 赛道采样点集中在弹珠周围 ~16px 半径内，相邻点的 cache line 大概率已在 L1/L2
- 100Hz 频率下，连续 tick 的弹珠位置相近，cache 仍然热

### 2.6 Power Management Task (Core 0, prio 1)

```
状态机: ~16 bytes (pm_state_t + 计时器)
```

缓存占用可忽略不计。

## 三、缓存压力汇总

| 组件 | L1 D 压力 | L2 压力 | 备注 |
|------|-----------|---------|------|
| Camera | 无 | 无 | DMA 直接 PSRAM |
| Display/LVGL | 中 | **中** | 字型/样式缓存 + 帧缓冲 |
| IMU | 无 | 无 | 全部 < 1KB |
| **游戏渲染** | **高** | **高** | 819KB 流式写，L2 被反复覆盖 |
| 边缘检测 | 中 | 中 | 128K 像素 × 5 遍，顺序扫描 |
| **YOLO 检测** | **高** | **极高** | 2.5MB 权重，彻底冲毁 L2 |
| Physics | 低 | 低 | 弹珠局部性优秀 |
| Power Mgmt | 无 | 无 | 忽略不计 |

### 两核同时运行时的 cache 竞争

最坏场景：**游戏进行中**（Core 1 渲染 30fps + Core 0 LVGL 30fps）

```
时间 →
Core 0 LVGL:   [字体渲染][缓冲交换][触摸轮询] ...
Core 1 Render:  [地板填色][瓦片blit][物体blit][弹珠] ...
L2 状态:        共享 256KB 被两核交替驱逐，有效容量 < 128KB/核
```

L2 的 256KB 由两个核竞争，每一方的有效 L2 远低于 256KB。游戏渲染的 819KB 流式写会不断驱逐 Core 0 LVGL 的字型缓存，导致 LVGL 反复从 PSRAM 重新加载。

## 四、内存分配策略分析

当前 `CONFIG_SPIRAM_MALLOC_ALWAYSINTERNAL=16384` 意味着：

| 分配大小 | 位置 | 经过 Cache？ |
|----------|------|-------------|
| < 16KB | 内部 SRAM (L2MEM) | 经过 L1 Cache (write-back) |
| ≥ 16KB | PSRAM | 经过 L2 + L1 Cache |

关键缓冲分配位置：

| 缓冲 | 大小 | 位置 | Cache 行为 |
|------|------|------|-----------|
| 相机帧 (mmap) | 1MB × 2 | PSRAM | DMA 写入，CPU 不读 |
| 预览双缓冲 | 640KB × 2 | PSRAM | L2 cacheable |
| 游戏双缓冲 | 819KB × 2 | PSRAM | L2 cacheable |
| 边缘检测缓冲 | ~128KB × 5 | PSRAM | L2 cacheable |
| YOLO 模型 | ~2.5MB | PSRAM | L2 cacheable |
| 赛道栅格 | ~100KB | PSRAM | L2 cacheable |
| 弹珠状态 | ~100B | 内部 SRAM | L1 cacheable |
| IMU 卡尔曼 | ~200B | 内部 SRAM | L1 cacheable |
| 任务栈 (5×) | ~28KB 合计 | 内部 SRAM | L1 cacheable |

**内部 SRAM 预算紧张**：`MALLOC_CAP_INTERNAL` 可用总量 ~768KB(L2MEM) - 保留(32KB) - 代码段 - 数据段 - 任务栈(~28KB)。小对象分配竞争内部 SRAM。

## 五、可优化项（按收益排序）

### 5.1 游戏渲染：帧缓冲使用写合并 (Write-Combine) ★★★

**问题**：819KB 帧缓冲以 write-back 模式写入。每次 CPU 写一个 pixel，先要读取整个 128B cache line（write-allocate），修改 2B，再标记脏。一帧 640×640 = 409,600 次 write-allocate = 52MB 额外的 PSRAM 读流量。

**方案**：对纯写入的帧缓冲，在写入前调用 `esp_cache_msync()` 将对应地址范围标记为待写回，或使用非缓存映射。ESP-IDF 对 DMA 缓冲有类似支持。

**预期收益**：减少 ~50% 的渲染 PSRAM 带宽消耗。

### 5.2 瓦片/精灵数据预加载到内部 SRAM ★★

**问题**：瓦片精灵 (~512KB) 和物体精灵 (~128KB) 大于 L2(256KB)。sprite_blit 函数随机访问精灵数据时，频繁 PSRAM miss。

**方案**：渲染每帧前，将当前可见的精灵复制到内部 SRAM 的临时缓冲中（利用 768KB L2MEM 的空闲部分）。

**预期收益**：消除精灵 blit 的 PSRAM 随机访问延迟。

### 5.3 YOLO 推理时挂起 LVGL 渲染 ★

**问题**：YOLO 推理 200-500ms 期间，LVGL 仍在 Core 0 渲染，两者同时竞争 L2 和 PSRAM 带宽。

**方案**：检测开始前暂停 LVGL（`lv_timer_pause()`），检测完成后恢复。

**预期收益**：检测速度提升 ~15-20%，但不影响功能（检测期间用户在看进度条）。

### 5.4 L2 Cache 分区锁定 (Cache Lock) ★

**问题**：双核互相驱逐对方的有用数据。

**方案**：ESP32-P4 支持 L2 Cache freeze（`CONFIG_SOC_CACHE_FREEZE_SUPPORTED=y`）。可将 L2 的一部分（如 128KB）锁定给 Core 0 的 LVGL 字型/样式缓存，防止被 Core 1 的帧缓冲流式写驱逐。

**预期收益**：LVGL 字体渲染不因游戏渲染而频繁 miss，帧率更稳定。

### 5.5 精灵数据用 `__attribute__((aligned(128)))` 对齐 ★

**问题**：精灵像素数组可能跨 cache line 边界（128B），一次 blit 操作波及 2 个 cache line。

**方案**：所有 sprite 数据结构的 pixels 成员按 L2 cache line 128B 对齐。

**预期收益**：轻微减少精灵访问的 cache line 加载数。

## 六、总结

| 维度 | 当前状态 | 评级 |
|------|---------|------|
| Core 0 缓存利用率 | LVGL 渲染竞争 L2，但 IMU/Camera 无压力 | 良好 |
| Core 1 缓存利用率 | 游戏渲染 streaming write 模式浪费 L2 | **需优化** |
| YOLO 检测 | 模型 >> 所有 cache，彻底冲毁 L2 | **已知限制** |
| 跨核 L2 竞争 | 游戏时两核同时渲染，互相驱逐 | **需优化** |
| 内部 SRAM | 小对象在 SRAM 命中好（IMU/Physics） | 良好 |
| PSRAM 带宽 | 渲染+检测+LVGL 同时访问 16MB PSRAM | 压力大 |

**当务之急**：5.1（帧缓冲写合并）改动最小，收益最大。5.2 和 5.4 是中期优化。5.3 和 5.5 是低优先级微调。
